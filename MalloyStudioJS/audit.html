1. Codebase audit

Current app structure

MalloyStudio is a Windows-only Qt 6 / C++17 desktop application built with CMake. Layout is a single main window (MainWindow) hosting a Preview widget and a constellation of QDockWidget-based panels (Scenes, Sources, Audio Mixer, Inspector, Controls Bar). The codebase is well-factored into model / capture / audio / recording / project / ui layers; nothing here is "bad" — it just stops at the OBS-equivalent surface area.

LayerFiles (representative)What it owns
model/Source, SceneItem, Scene, SceneCollection, FilterEffectPure data + undo + JSON I/O. Source library (shared refs), per-item transform + filter chain.
capture/DxgiCapture, WindowCapture, WindowCaptureSession, WasapiCapture, CaptureControllerWorker-thread capture engines + a controller that reconciles active sessions to scene state.
audio/AudioInput, AudioControllerOwns WASAPI workers, per-source reconcile, the VU bus.
recording/EncoderRegistry, EncoderPipeline, MediaController, StreamingPipeline, StreamSettings, OutputSettings, Ring*Sourceffmpeg subprocess, video stdin + named-pipe audio, replay buffer, RTMP streaming, hardware encoder selection.
project/ProjectDocumentAtomic load/save of .malloy.json via QSaveFile.
input/HotkeyManagerGlobal hotkeys (mute, record toggle, scene switch, etc.).
ui/16 files — Preview, ControlsBar, Inspector, AudioMixer, Scenes/Sources panels, picker dialogs, Hotkeys, VuMeterAll visible chrome. This is where ~95% of the redesign lands.

Existing design system

There isn't one yet, really. resources/styles/dark.qss is the only stylesheet — 64 lines, single-purpose: it normalizes a few objectName-based widgets (#emptyState, #recordTimer, #streamTimer, #hintText, #typeAnnotation) that were previously styled via inline setStyleSheet(). The comment in the file is honest: "Migrated from inline calls scattered across the UI."

There's no token system, no documented type scale, no spacing scale, no shared button/input classes. The recording-red (#e05050) and stream-orange (#e08030) colors are hardcoded in the QSS. Icons live in resources/icons/ but the set is incomplete — many panels rely on text-only buttons.

Existing components (Qt widgets)

  • Keep, light reskin: PreviewWidget, VuMeter, AudioMixerPanel, ScenesPanel, SourcesPanel, InspectorPanel, ControlsBar.
  • Refactor: the picker dialogs (MonitorPickerDialog, WindowPickerDialog, MicrophonePickerDialog) are functional but visually inconsistent — every dialog reinvents its row layout. Extract one PickerDialogBase and reuse.
  • Replace: HotkeysDialog is a single-column form per assignment. Move to a searchable, two-pane editor (categories left, bindings right) when v0.2 lands.
  • Net-new for v0.1: Dashboard, Render Queue, Clips Library, Media Library, Project Manager, Onboarding, AI Lab placeholder.
  • Net-new for v0.2+: Timeline editor and all of its surfaces (media bin, transport, inspector tabs, effect cards, ruler, track headers, clip widgets, snapping/markers).

Strengths to protect

Subsystem boundaries are clean

Model / capture / audio / recording / project / ui split is exactly what you want when adding a Vegas-style editor on top: the editor doesn't need to know anything about DXGI.

Snapshot undo + atomic save

SceneCollection snapshot-based undo and QSaveFile for .malloy.json means project data is hard to corrupt. Preserve this; the editor's timeline state should be undoable through the same mechanism.

Replay buffer + hardware encoders

NVENC / QSV / AMF via EncoderRegistry, plus the in-memory ring buffer. This is the "Clip" feature creators expect, already built. Surface it more prominently on the Dashboard.

Tests for the model layer

12 model tests is a real foundation — JSON migration, undo coalescing, capture reconciliation. Adding editor / timeline tests later will be straightforward.

Weaknesses to fix

No design tokens, no typography scale

QSS is fragmented. Hardcoded colors (#e05050, #e08030, #888, #9a9a9a) appear with no semantic name. Action: introduce a tokens.qss with CSS-variable-equivalent (Qt has qApp->property()) and propagate.

Recording state isn't ambient

Today the recordTimer shows in the ControlsBar only. Once the window is wider than the bar, you can lose track of it. Action: add a persistent status footer (mirrored from the prototype) that lives at the bottom of the main window regardless of dock layout.

Inspector "no properties" page

QLabel#filterEmptyPage is the only empty state in the entire UI. Every list (scenes, sources, future media bin) needs a teaching empty state — not just gray text.

Source-type Add menu is the only discovery path

New users land in a window with empty Scenes + Sources panels and don't know to click "+" then pick a type. Action: first-launch onboarding that runs device detection and inserts a starter scene.

Dialogs over modes

OutputSettingsDialog, StreamSettingsDialog, HotkeysDialog, three picker dialogs — all modal popups that interrupt the canvas. Action: a unified Settings workspace; pickers become inline panels where they can.

No keyboard discoverability

HotkeyManager exists but the only place to see bindings is the Hotkeys dialog. Action: tooltips display the shortcut for every actionable button; a ⌘K command palette surfaces all actions by name.

No project hub / dashboard

The app opens directly into a blank capture canvas. Creators want a "where am I, what am I working on" view. Action: Dashboard as the default workspace.

Filename: recording/MediaController.cpp is misnamed

It owns the encoder pipeline, not media. Suggest renaming to RecorderController to free up the "Media" namespace for the editor's bin.

2. Product understanding

MalloyStudio is a local-first creator workstation for Windows built on Qt + the Windows capture stack. It targets one user — a solo creator who records gameplay or screen content, sometimes streams, then edits and publishes — who is currently using two apps (a capture app + an NLE) and wants one.

The product is structured around five durable verbs:

  • Capture — record, stream, clip. (Mostly built.)
  • Organize — projects, media library, clips library.
  • Edit — timeline, multi-track, transitions, text, effects, export.
  • Render — background queue with retries.
  • Augment — eventually AI helpers across all of the above.

3. Questions still open

  1. Editor depth: Are we matching Vegas/Premiere features 1:1 long-term, or is the editor "good enough to finish a stream highlight" — i.e., simpler cut-and-assemble for fast turnaround?
  2. Cross-platform: The capture stack is Windows-only by design. Is macOS/Linux ever on the roadmap, or do we commit fully to a Win32 creative app?
  3. Plugins: Will MalloyStudio support OBS-style plugins? If yes, the source-type registry needs to be extensible from the start.
  4. AI hosting: Local Whisper/CLIP is the plan, but transcription on a 2-hour stream takes time. Optional cloud fallback?
  5. Stream destinations: Multi-stream out (Twitch + YouTube simultaneous via a service like Restream) — first-party or pass-through?
  6. Auto-clip detection: Audio-driven (loud reactions) or game-event driven (overlay APIs / process hooks)?
  7. Project format: Should the editor's timeline live in .malloy.json (one project = one file) or split into .malloyproj bundles?
  8. Window size minimum: Is 1280×720 still acceptable as the floor, or are we committing to 1440×900 for the redesign?

5. Core user workflows

  1. Cold-start to recording — first launch → onboarding (device detect) → Dashboard → "Start Recording" → recording active in Recording Studio. Target: under 60 seconds from app icon to red-dot.
  2. Stream-while-recording — Dashboard → "Go Live" → preflight check (key valid? bandwidth? scene set?) → live + recording simultaneously. Replay buffer stays armed.
  3. Clip during gameplay — global hotkey ⌥⌘C saves last 30 s from the replay buffer → toast confirms with a "Open in Editor" CTA → clip appears in Clips Library tagged with current project.
  4. Highlight reel — Clips Library → select N clips → "Open in Editor" → editor opens with selected clips already on the timeline → trim, add titles, export.
  5. Render & publish — Editor → "Export" → render queued (background, doesn't block recording) → completed render shown in Render Queue with "Upload to YouTube" / "Open folder" actions.
  6. Resume after crash — every workspace saves layout + transient state. App boots back into the same workspace with autosave-recovered project.

Required data
Current project (or null), recent recordings (last 8), recent projects (last 8), recent clips (last 12), active+pending+completed renders, system health probes (devices, encoder, storage, stream key).
Empty state
If no project: hero card becomes a "Create your first project" CTA; system status still shows device readiness; Quick Actions remain available.
Loading state

Primary actions

Keyboard

6.7 Project Manager v0.1

Card grid of .malloy.json projects, with cover thumbnail (last preview frame), size, last edited, status (Open / Active / Draft / Final / Archived). Actions: Open, Duplicate, Archive, Reveal in Explorer.

6.8 Render Queue v0.1

Tab strip: Active / Pending / Completed / Failed. Active jobs show animated progress + ETA. Failed jobs show ffmpeg exit code + log link; "Retry" re-enqueues. Completed jobs persist with "Open folder" / "Upload" / "Show in finder".

6.9 Settings v0.1

Two-pane: groups (left), content (right). Groups: General, Recording, Streaming, Video, Audio, Hotkeys, Storage, Performance, Appearance, Accounts, Experimental, AI placeholders. Consolidates the current OutputSettingsDialog + StreamSettingsDialog + HotkeysDialog.

6.10 Onboarding v0.1

7 steps, never longer than 90 seconds: welcome → workflow choice → folder choice → device detect → quality → density → done. Skippable. Re-launchable from Settings → General → "Run setup wizard".

6.11 AI Lab v0.3

v0.1 ships the placeholder screen — feature cards labelled "Soon". v0.3 unlocks them progressively (subtitles first, then clip detection, then chapters, etc).

7. Design system

Visual direction

Studio-grade dark. Linear / Arc-influenced. Tightly-scoped neutral gray surfaces with a single accent color (blue by default; user-swappable). Recording red is the only other "loud" color — reserved for record/live states. Monospace (Geist Mono) carries all technical readouts (timecodes, dB, bitrates, file sizes); sans (Geist) carries everything else.

State badges

Recording / Live: red dot with a 1.4s pulse. Rendering: spinner. Idle: gray static dot. Degraded streaming: amber chip. Color is never the only signal — a label always accompanies the dot (REC, LIVE, IDLE, RENDERING). This satisfies the "states beyond color alone" accessibility requirement.

15. Risks & blockers

  • Timeline performance — Qt's QGraphicsScene can struggle with 1000+ clip widgets. Plan: render timeline directly via QPainter on a custom QWidget, virtualize clips outside the viewport.
  • Decoder cost — playback of multiple 4K streams in the preview monitor needs hardware decode (D3D11VA via ffmpeg). Single 1080p60 is fine; ramping up to 4-stream multicam needs investigation.
  • Project format migration — adding the timeline to .malloy.json requires a v3 schema. The model layer already has a v1→v2 migration; reuse that pattern.
  • Window capture chrome — README admits client-area only. DXGI per-window via WGC (Windows.Graphics.Capture) is the right answer; requires Windows 10 1903+ which we already require.
  • Stream key storage — never plain-text in the project file. Use Windows Credential Manager via QtKeychain.
  • Replay buffer + memory — currently in-memory ring. 30 s @ 1080p60 is ~8 GB raw. Confirm we're storing encoded frames (we are; just documenting).
  • AI on big files — a 2-hour 1080p60 stream is 14 GB. Whisper transcription with Word-level timestamps is many minutes. The UI must communicate "this takes time" gracefully and allow cancel/resume.
  • QSS reach — Qt stylesheets can't do everything (compound selectors, container queries). Some custom widgets will need to do their own painting. Pre-plan the boundary.
  • One-developer bandwidth — the v0.2 editor is ~3 months of work for one person. Consider shipping v0.1 as a 1.0 release, marketing it as a capture + clips workstation, and treating the editor as a separate v2.0.

16. Immediate next steps

  1. Answer the 8 open questions above. Especially editor depth, plugin support, and cross-platform — those determine the architecture.
  2. Adopt the token system — port tokens.css into a tokens.qss + qApp palette. Replace hardcoded hex values across the existing UI (one PR per panel).
  3. Land AppShell + IconRail + StatusBar. Keep the current MainWindow capture surface inside the new "Recording" workspace. No functional change, only chrome.
  4. Build Dashboard skeleton. Wire it to live data: SceneCollection for current project, RecorderController for recordings list, ProjectRegistry (new) for projects.
  5. Extract PickerDialogBase and migrate the three picker dialogs.
  6. Begin Clips/Media/Projects workspaces in parallel — they share the same "left filter + main content" pattern.
  7. Set up a feature-flag system so we can ship the new shell with the old workspaces fallback-able. Big releases like this are best landed behind a flag.
  8. Write three more model tests: ClipsRegistry roundtrip, RenderQueue retry semantics, ProjectRegistry rescan idempotence.
